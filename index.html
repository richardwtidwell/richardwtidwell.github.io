CS 499 Module Two Journal – Code Review Plan

Code review is a structured peer examination of a change set that checks design, correctness, readability, security, and maintainability before integration. It spreads knowledge across the team, finds defects earlier than testing alone, and enforces shared engineering standards, lowering rework and operational risk (Google, n.d.; SmartBear, n.d.; OWASP Foundation, n.d.; Winters, Manshreck, & Wright, 2020).

My recorded review keeps PRs small, follows a visible checklist, and addresses three lenses:

  • Software engineering & design: Validate separation of concerns, typing, validation, error handling, logging, tests, and documentation.
  • Algorithms & data structures: Reconfirm problem statements, Big-O claims, boundary conditions, and benchmark evidence.
  • Databases: Inspect schemas, indexes, validation rules, and least-privilege access while logging follow-up actions.

Algorithms & Data Structures

Search acceleration via an LRU cache, prefix trie, server-side pagination, and indexed MongoDB queries to keep latency predictable.

Read Milestone Three narrative

The enhanced artifact is the Travlr search and listing subsystem. It exposes Mongo-backed endpoints consumed by the admin UI. For this milestone I focused on measurable latency improvements at the API layer, introducing:

  • A capacity-bound LRU cache with TTL and instrumentation to reuse identical queries.
  • A case-insensitive prefix trie powering typeahead for trip codes and names.
  • Server-side pagination and compound sorting so work scales with page size rather than collection size.
  • Repository/service abstractions with stable cache keys, periodic trie refresh, input validation, consistent error envelopes, and rate limiting.

Tests cover the LRU and trie using node:test, while benchmark scripts capture trie build times and cache throughput. The work addresses Outcome 3 by selecting structures with known complexity, Outcome 4 through layered design plus tests/benchmarks, and Outcome 5 via validation, bounded caches, and abuse-resistant limits.

Reflection: I learned to favour designs with predictable cost and to pair indexing with in-memory assistance only when the working set can stay hot. Stable ordering, staleness, and worst-case query sweeps were mitigated with lexical traversal limits, periodic refresh, strict validation, and rate limiting.

Databases

Reservation subsystem enabling atomic bookings, reservation audit trails, and JWT-protected occupancy analytics.

Read Milestone Four narrative

The database enhancement focuses on extending the Travlr API to support reservations end to end. Key additions include embedded reservation documents (customer name/email, party size, notes, timestamps), capacity-aware booking logic, and JWT-protected reporting endpoints.

  • Enhancements: Reservation services (`reservationService.bookTrip` / `getOccupancyReport`), Zod validation, structured errors, and synced Angular service models.
  • Evidence & testing: Curl sessions capture successful bookings and occupancy summaries, while `src/test/bookingValidation.test.ts` exercises the new validator.
  • Outcomes covered: Outcome 3 (aggregation pipelines and deterministic capacity checks), Outcome 4 (layered design, tests, docs), Outcome 5 (JWT protection, strict validation, idempotent capacity checks).

Reflection: Aligning schema design with real-world invariants (remaining seats, audit history) while keeping controllers thin reinforced the need for clear seams between services and data. Future work could include cancellation flows, reporting UI, and regression tests against dockerised MongoDB.
